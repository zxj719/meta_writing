## 一、项目概述

**meta_writing** 是一个多 Agent 中文网络小说自动生成系统。核心目标：通过多个专业化 AI Agent 协作，生成百万字级别的长篇网络小说，同时保持人物、剧情、风格、主题的一致性。

**质量标准：** 网文读者无法分辨是 AI 生成的。
**目标平台：** 番茄小说、起点中文网等网络小说平台。

## 二、Office Hours 头脑风暴阶段

### 2.1 问题定义

当前 LLM 在单章级别可以生成合格的网文散文，但在 100+ 章、百万字的规模下完全无法维持一致性：
- 人物声音模糊
- 前期铺垫的伏笔后期遗忘
- 主题一致性漂移
- 世界观规则自相矛盾

番茄小说上 AI 生成内容同比激增 13 倍，但全是粗制滥造，没有任何方案解决一致性问题。

### 2.2 三种方案对比

| 方案 | 描述 | 工作量 | 风险 | 完整度 |
|------|------|--------|------|--------|
| A: Story Bible + 流水线 | 线性流程，简单 Python 脚本编排 | S | 低 | 6/10 |
| **B: 多 Agent 集成 + 共享记忆** | 专业化 Agent + 共享 Story Bible 记忆层 | M | 中 | **9/10** |
| C: 生活世界模拟 | 角色作为独立 Agent，涌现式叙事 | L | 高 | 7/10 |

**最终选择：方案 B**——架构完整、可扩展，是通往方案 C（10x 愿景：活的故事世界）的自然演进路径。

### 2.3 Agent 架构（5 个专业 Agent + 1 个编排器）

1. **Orchestrator（编排器）**——管理整体工作流、路由任务、维护流水线状态
2. **Planner Agent（策划）**——生成下一章大纲，提供 2-3 个剧情分支选项，人类选择走向
3. **Writer Agent（写手）**——根据选定大纲 + Story Bible + 近期章节上下文，生成完整章节散文
4. **Continuity Agent（连续性检查）**——验证章节与 Story Bible 的一致性：人物状态、关系、时间线、世界规则、伏笔审计
5. **Style Agent（风格检查）**——叙事声音、POV 一致性、网文惯例、AI 味检测
6. **Theme Agent（主题检查）**——核心主题追踪、角色弧线对齐、价值观一致性

### 2.4 写作方法论集成

系统将成熟的网文创作方法论内嵌到各个 Agent 的行为中（参考 `skills_novel-writing-expert-v2.md`）：

| 方法论 | 对应 Agent |
|--------|-----------|
| 三幕式结构、爽点分布、钩子布局、五层世界揭示 | Planner |
| 五感描写、对话四功能、长短句节奏 | Writer |
| 角色动机三角验证、行为逻辑("为什么?")、五阶段成长弧线 | Continuity |
| 节奏公式、钩子强制、对话质量、AI 味检测 | Style |
| 五层世界架构对齐、角色弧线阶段追踪、核心主题一致性 | Theme |

**七大核心框架：**
1. **一句话核心（Hook）**——故事开始前必须有一句话概括核心吸引力
2. **五层世界架构法**——表层世界 → 规则层 → 禁忌层 → 真相层 → 本质层
3. **角色核心三角形**——欲望 × 能力 × 阻碍
4. **动机类型矩阵**——生存/情感/利益/使命/好奇心
5. **爽点布局**——小爽点（章内）/ 中爽点（副本中期）/ 大爽点（弧线高潮）
6. **钩子技术**——悬念/冲突/情感/反转四种钩子
7. **节奏公式**——紧张 → 短暂缓解 → 更大紧张 → 高潮 → 余韵

### 2.5 共享记忆层：Story Bible

**结构化状态（YAML 文件）：**
- `story_core.yaml`——一句话核心、体裁、目标爽点类型、五层世界架构
- `characters/`——每个角色一个文件：外貌、性格、知识状态、情感状态、关系、目标、位置、核心三角、动机类型、成长弧线阶段
- `timeline.yaml`——有序事件 + 章节引用
- `world_rules.yaml`——魔法体系、科技水平、地理、社会结构
- `foreshadowing.yaml`——伏笔设置/回收对，状态：`planted` | `reinforced` | `paid_off` | `abandoned`
- `pacing.yaml`——爽点时间表、钩子布局日志、张力曲线追踪
- `chapter_summaries/`——每章结构化摘要

**向量存储（语义检索）：**
- ChromaDB + BGE-M3（本地中文优化嵌入）
- 所有生成章节按场景分块（~500-1000 字/块）嵌入
- 支持"找到 X 发生的那一章"查询

### 2.6 每章工作流

```
1. Orchestrator 加载当前 Story Bible 状态
2. Planner Agent 生成 2-3 个剧情分支选项
3. 人类选择分支（可修改/组合）
4. Writer Agent 起草完整章节
5. 并行审查：Continuity + Style + Theme Agent
6. 发现问题 → Writer 修改（最多 3 轮迭代）
   - 3 轮后仍未解决 → 上报人类，展示剩余问题的 diff
   - 审查 Agent 反馈冲突 → Orchestrator 展示双方观点，人类裁决
7. 人类审查最终稿（批准 / 退回附注释 / 行内编辑）
8. Orchestrator 原子提交：章节文本 + Story Bible 更新 + 伏笔更新 + 章节摘要
9. 重复
```

### 3.7 失败模式与恢复

- **API 中途故障：** 指数退避重试（3 次），持久化部分状态，允许从检查点恢复
- **Story Bible 损坏：** git 回滚到最后已知良好提交，每章提交是原子的
- **人工编辑引入新信息：** Orchestrator 触发 Story Bible 调和——Continuity Agent 重新验证并提取新实体/事件
- **追溯修正（Retcon）：** 人类标记 retcon，更新 Story Bible，生成 retcon 日志供后续章节参考
- **限流保护：** 审查 Agent 错开 1 秒延迟，Orchestrator 实现请求队列

### 8.3 关键创作发现与决策

**世界观规则执行：**
- 微感只能读物体的物理痕迹（声音/温度/磨损），绝不能读心
- Planner 曾生成葬礼场景中"他的女儿在想……"的读心术描写，必须在 Writer prompt 中硬性纠正
- 写作铁律："X记得Y"句式禁止，物体没有意图（不写"在说话""在等"），让读者自己连线

**风格基调确立：**
- 冷静、克制、有距离感的第三人称
- "沉默代替抒情"——用动作/触感/环境音传递情感，不写"她懂那种孤独"
- "她不知道"全文限制 ≤ 3 次
- 比喻密度控制：相邻 500 字内"像"不超过 2 次

**生成策略演进：**
- ❌ 修订式（提供原文让模型修改）→ 模型倾向复制原文
- ✅ 新生成式（只给详细大纲，不给原文）→ 质量更高
- ✅ 分步扩写式（先生成骨架，再在指定位置插入细节）→ 可控性最好
- 详细的场景级大纲（800-1200 字/场景）比简短大纲效果显著更好

### 8.4 系统增强：三层文风防线

为解决"每章都需要人工逐行修复直白解读"的问题，实现了三层自动化防线：

**第1层：Writer System Prompt 硬规则（生成前）**
- 在 `writer.py` 的 `WRITER_SYSTEM_PROMPT` 中新增"微感描写铁律"
- 明确禁止"X记得Y"、拟人化、读心术，附正反示例

**第2层：StyleLinter 正则扫描（生成后，零成本）**

**第3层：Continuity Agent 文风审查（生成后，LLM 成本）**
- `continuity.py` 新增第 8 项检查"微感描写文风"

**Pipeline 集成：** StyleLinter 在 Continuity review 之前运行，如有 ERROR 级问题即使 Continuity 通过也触发 Writer revision，两份 feedback 合并发送。

## 九、反思：Human-in-the-Loop 瓶颈分析与改进方向

### 9.1 当前问题

每章的实际生成流程严重偏离设计的自动化 pipeline：

```
设计流程：Plan → Write → Auto-Review → Auto-Revise (≤3轮) → Human Review → Commit
实际流程：Plan → Write → Human 发现质量不达标 → 手动重写/逐行修复 → 扩写 → 再逐行修复
```

**Human-in-the-loop 介入点统计：**
- Planner JSON 解析失败 → 人工修复 JSON（每次）
- Writer 首稿字数不足 → 人工编写扩写脚本
- Writer 违反微感描写规则 → 人工逐行修复
- Writer 术语感/解释过度 → 人工逐行优化
- 总计每章需要 2-3 轮人工干预，每轮 10-30 分钟

### 9.2 根因分析

| 问题 | 根因 | 当前状态 |
|------|------|---------|
| **Planner JSON 解析失败** | MiniMax API 返回未闭合 markdown code block | 每次手动修复 |
| **首稿字数不足** | Writer 单次生成上限 ~5000-8000 字，远低于 10000-14000 目标 | 每次手动写扩写脚本 |
| **微感"直白解读"反模式** | Writer prompt 不够严格，模型倾向于解释 | ✅ 已通过三层防线部分解决 |
| **术语感/过度科学化** | 模型在描写物理细节时退化为论文风格 | 每次手动优化 |
| **Continuity Agent 未实际运行** | 当前绕过 pipeline 使用独立脚本生成 | 未自动化 |
| **Style/Theme Agent 未实现** | Post-MVP，但风格问题是当前最大人工负担 | 未实现 |

### 9.3 改进方案

#### P0：立即可做（减少 50%+ 人工干预）

**1. Planner JSON 鲁棒性修复**
- 在 `planner.py` 的 `_parse_response` 中增加多重 fallback：尝试 `json.loads` → 正则提取 `{...}` → 修复未闭合引号/括号 → 最后才 fallback 到单分支
- 可调用：直接修改 `meta_writing/agents/planner.py`

**2. Writer 两阶段生成（骨架+扩写自动化）**
- 第一阶段：Writer 生成骨架（~5000-8000 字）
- 第二阶段：自动检测字数不足时，Orchestrator 调用第二轮 Writer 扩写
- 扩写 prompt 自动生成：基于骨架中的场景分割点，自动标注"此处需要扩充"
- 可调用：修改 `meta_writing/orchestrator.py` + `meta_writing/agents/writer.py`

**3. Writer Prompt 中的反面示例库**
- 将已积累的反模式（"X记得Y"、术语化、过度解释）编入 system prompt 的 few-shot 示例
- 每次人工修复后，将修复前→修复后的 pair 存入 `story_data/style_examples.yaml`
- Writer 生成时自动加载这些 pair 作为 negative/positive 示例
- 可调用：新建 `story_data/style_examples.yaml` + 修改 `writer.py`

#### P1：近期实现（进一步减少 30% 人工干预）

**4. Style Agent 实现**
- 不需要等 Post-MVP——当前最大人工负担恰好是风格问题
- 专注于：术语感检测、解释过度检测、比喻密度检测、"沉默传情"原则验证
- 比 Continuity Agent 更重要，应优先实现
- 可调用：`/plan-eng-review` 做技术设计，然后实现

**5. 参考章节自动注入**
- Writer 生成时自动注入最近 1-2 章的"最佳段落"作为风格锚点
- 从已通过人工审核的章节中自动提取高质量片段（微感描写、对话、留白）
- 让模型"模仿"已有好的风格，而非仅靠 prompt 规则约束
- 可调用：修改 `compressor.py` + `writer.py`

**6. 自动化回归测试**
- 每次 StyleLinter 规则库更新后，自动对所有已有章节重新扫描
- 将已完成章节作为"黄金标准"，新生成的章节必须通过与已有章节相同的质量阈值
- 可调用：新建 `tests/test_chapter_quality.py`

#### P2：中期优化（系统性解决）

**7. 回到 Pipeline 轨道**
- 当前绕过 orchestrator 使用独立脚本（`gen_ch4.py`, `gen_ch5.py`），导致所有自动审查机制失效
- 应将独立脚本中的 system prompt 和 outline 质量提升后，合并回 `orchestrator.py` 的标准流程
- 目标：`meta-writing generate` 一条命令跑完 Plan→Write→Review→Revise→Commit

**8. MiniMax → Anthropic API 迁移评估**
- MiniMax 的 JSON 输出质量差是 Planner 反复失败的根因
- 评估切换到 Anthropic 原生 API（Claude Sonnet/Opus）的成本与质量提升
- 或者在 Planner 中使用 Anthropic，Writer 继续使用 MiniMax（混合策略）
